# Plan 1.2: SPEC Template Update + Plan-Checker Enforcement

## Objective
Update the SPEC template to support `[NEEDS CLARIFICATION]` markers, add Quality Requirements and Edge Cases sections, and update the plan-checker skill to reject specs with unresolved markers. This ensures no ambiguous spec can reach the planning stage.

## Context
- .gsd/DECISIONS.md (D-002, D-003, D-007)
- .gsd/references/spec-kit/templates/spec-template.md (their [NEEDS CLARIFICATION] pattern)
- .gsd/templates/spec.md (current template)
- .agents/skills/plan-checker/SKILL.md
- .agent/workflows/new-project.md (Phase 4 — Write SPEC.md section)

## Tasks

<task type="auto">
  <name>Update SPEC template with new sections</name>
  <files>.gsd/templates/spec.md</files>
  <action>
    Add the following to the existing spec template (ADDITIVE — do not remove existing sections):

    1. Add instruction block at top (after Status line):
       ```
       > ⚠️ **Clarification Rule**: When creating this spec, mark ALL ambiguities with
       > `[NEEDS CLARIFICATION: specific question]` inline. Do not guess. If the user
       > didn't specify something, mark it. All markers must be resolved before FINALIZED.
       >
       > 📐 **Separation Rule**: This spec defines WHAT to build and WHY.
       > - ✅ Focus on user goals, business requirements, constraints, success criteria
       > - ❌ Avoid HOW to implement — no tech stack, APIs, frameworks, code structure
       > - Technical choices belong in PLAN.md, not here
       ```

    2. Add "Quality Requirements" section (after Constraints, before Success Criteria):
       ```markdown
       ## Quality Requirements
       - **Error Handling**: [NEEDS CLARIFICATION: fail-fast / graceful degradation / hybrid?]
       - **Logging**: [NEEDS CLARIFICATION: structured JSON / plaintext? minimum log level?]
       - **Performance Targets**: [NEEDS CLARIFICATION: latency targets? throughput? p95/p99?]
       - **Security**: [NEEDS CLARIFICATION: auth method? data encryption? CORS policy?]
       ```
       These are placeholder examples showing the [NEEDS CLARIFICATION] pattern in action.

    3. Add "Edge Cases" section (after Success Criteria):
       ```markdown
       ## Edge Cases
       - What happens when [boundary condition]?
       - How does the system handle [error scenario]?
       - What if there are zero results? Maximum results?
       ```

    4. Add "Unresolved Questions" section at the very bottom:
       ```markdown
       ## Unresolved Questions
       <!-- Auto-generated by /stress-test. Lists all [NEEDS CLARIFICATION] markers. -->
       <!-- This section must be EMPTY before spec can be marked FINALIZED. -->
       ```

    Reference: .gsd/references/spec-kit/templates/spec-template.md lines 93-97 for their marker syntax.
  </action>

    grep -q "Unresolved Questions" .gsd/templates/spec.md && \

<task type="auto">
  <name>Update plan-checker to reject unresolved markers</name>
  <files>.agents/skills/plan-checker/SKILL.md</files>
  <action>
    Add a new check to the plan-checker skill:

    **New Rule: Spec Clarity Gate**
    Before checking plans, verify SPEC.md has no unresolved clarification markers:
    ```bash
    if grep -q "\[NEEDS CLARIFICATION" .gsd/SPEC.md; then
        echo "FAIL: SPEC.md has unresolved [NEEDS CLARIFICATION] markers"
        grep -n "\[NEEDS CLARIFICATION" .gsd/SPEC.md
        echo "Resolve all markers before planning. Try /stress-test to review."
        exit 1
    fi
    ```

    Add this as the FIRST check in the plan-checker flow (before validating plan files).
    
    Also update the checker's documentation to explain this rule and reference D-002.
  </action>
  <verify>
    grep -q "NEEDS CLARIFICATION" .agents/skills/plan-checker/SKILL.md && \
    echo "PASS: Plan-checker rejects unresolved markers"
  </verify>
  <done>
    - Plan-checker has spec clarity gate as first check
    - Rejects specs with any [NEEDS CLARIFICATION] markers
    - Shows line numbers of unresolved markers
    - Suggests /stress-test for review
  </done>
</task>
